# AionUI 集成到 VS Code - 需求文档

## 1. 项目概述

### 1.1 目标
将 AionUI（一个完整的 Electron + React AI 聊天应用）集成到 VS Code 源码中，使其能够通过命令独立弹出，而不复用 VS Code 原生组件。

### 1.2 参考架构
参考 `vs/sessions` 模块的独立窗口架构，但采用更简单的 Electron BrowserWindow 方式。

### 1.3 核心原则
- **独立性**：AionUI 保持完全独立，不依赖 VS Code UI 组件
- **可维护性**：AionUI 可以独立开发和构建
- **集成性**：通过 VS Code 命令/按钮唤起
- **通信能力**：可选的 IPC 通信支持

## 2. 功能需求

### 2.1 启动方式

#### FR-1: 命令行启动
- **需求**：支持通过命令行参数 `--aionui` 启动
- **示例**：`code --aionui`
- **优先级**：P0（必须）

#### FR-2: 命令面板启动
- **需求**：在 VS Code 命令面板中提供 "Open AionUI Window" 命令
- **命令 ID**：`workbench.action.openAionUIWindow`
- **优先级**：P0（必须）

#### FR-3: 状态栏按钮（可选）
- **需求**：在状态栏添加 AionUI 图标按钮
- **位置**：状态栏右侧
- **优先级**：P1（建议）

### 2.2 窗口行为

#### FR-4: 独立窗口
- **需求**：AionUI 在独立的 Electron BrowserWindow 中运行
- **特性**：
  - 独立的窗口进程
  - 可以独立于 VS Code 主窗口存在
  - 支持最小化、最大化、关闭
- **优先级**：P0（必须）

#### FR-5: 窗口配置
- **默认尺寸**：1200x800
- **最小尺寸**：800x600
- **标题**：AionUI - AI Assistant
- **图标**：使用 AionUI 的应用图标
- **优先级**：P0（必须）

#### FR-6: 单例模式
- **需求**：同一时间只能打开一个 AionUI 窗口
- **行为**：如果已存在窗口，则聚焦现有窗口
- **优先级**：P0（必须）

### 2.3 构建集成

#### FR-7: 独立构建保留
- **需求**：AionUI 保留独立构建能力
- **位置**：`extensions/aionui-main/`
- **构建命令**：保持原有的 `bun run build` 等命令
- **优先级**：P0（必须）

#### FR-8: VS Code 构建集成
- **需求**：VS Code 构建时自动构建 AionUI
- **集成点**：
  - 在 VS Code 的 gulp 构建流程中添加 AionUI 构建步骤
  - 将 AionUI 构建产物复制到 VS Code 输出目录
- **优先级**：P0（必须）

#### FR-9: 开发模式支持
- **需求**：开发时支持热重载
- **方案**：
  - 开发模式下加载 AionUI 的开发服务器（localhost:5173）
  - 生产模式下加载打包后的静态文件
- **优先级**：P1（建议）

### 2.4 通信能力（可选）

#### FR-10: 基础 IPC 通信
- **需求**：AionUI 可以与 VS Code 主进程通信
- **用例**：
  - 获取当前打开的文件路径
  - 获取当前工作区信息
  - 接收 VS Code 事件通知
- **优先级**：P2（可选）

#### FR-11: 文件操作
- **需求**：AionUI 可以请求 VS Code 打开文件
- **方法**：通过 IPC 调用 VS Code 的文件服务
- **优先级**：P2（可选）

## 3. 非功能需求

### 3.1 性能

#### NFR-1: 启动速度
- **需求**：AionUI 窗口应在 2 秒内显示
- **优先级**：P1（建议）

#### NFR-2: 内存占用
- **需求**：AionUI 窗口内存占用不超过 500MB
- **优先级**：P1（建议）

### 3.2 兼容性

#### NFR-3: 平台支持
- **需求**：支持 macOS、Windows、Linux
- **优先级**：P0（必须）

#### NFR-4: VS Code 版本
- **需求**：兼容当前 VS Code 源码版本
- **优先级**：P0（必须）

### 3.3 可维护性

#### NFR-5: 代码隔离
- **需求**：AionUI 代码与 VS Code 代码清晰分离
- **位置**：
  - AionUI 源码：`extensions/aionui-main/`
  - VS Code 集成代码：`src/vs/aionui/`
- **优先级**：P0（必须）

#### NFR-6: 文档完整
- **需求**：提供完整的集成文档
- **内容**：
  - 架构说明
  - 构建指南
  - 开发指南
  - 故障排查
- **优先级**：P0（必须）

## 4. 约束条件

### 4.1 技术约束
- AionUI 使用 Electron + React + Vite
- VS Code 使用 Electron + TypeScript + Gulp
- 需要保持两者的构建系统独立

### 4.2 架构约束
- 不修改 AionUI 的核心代码
- 最小化对 VS Code 核心代码的修改
- 遵循 VS Code 的代码规范和架构模式

### 4.3 兼容性约束
- 不影响 VS Code 的正常功能
- 不影响 `vs/sessions` 模块的运行
- 支持与 VS Code 同时运行

## 6. 风险与缓解

### 6.1 技术风险

#### 风险 1: 构建系统冲突
- **描述**：AionUI 使用 Vite，VS Code 使用 Gulp
- **影响**：可能导致构建失败或产物冲突
- **缓解**：
  - 保持两个构建系统完全独立
  - 使用脚本桥接两个构建流程
  - 明确定义产物输出路径

#### 风险 2: Electron 版本不兼容
- **描述**：AionUI 和 VS Code 可能使用不同版本的 Electron
- **影响**：可能导致运行时错误
- **缓解**：
  - 检查 Electron 版本兼容性
  - 必要时升级 AionUI 的 Electron 版本
  - 使用 VS Code 的 Electron 版本

#### 风险 3: 依赖冲突
- **描述**：AionUI 和 VS Code 可能有依赖冲突
- **影响**：构建或运行时错误
- **缓解**：
  - 保持依赖隔离
  - 使用独立的 node_modules
  - 明确依赖边界

### 6.2 维护风险

#### 风险 4: 双重维护负担
- **描述**：需要同时维护 AionUI 和集成代码
- **影响**：增加维护成本
- **缓解**：
  - 最小化集成代码
  - 清晰的代码边界
  - 完善的文档
